## <a name="Ingresso_altro_nodo"></a>Ingresso di un altro singolo nodo nella nostra rete

Ora assumiamo che il sistema ** giunga a distanza di rilevamento con la sua interfaccia di rete
per l'interfaccia di rete del sistema **.

Fra i compiti del modulo Neighborhood, esso aggiunge nel network namespace default la rotta diretta verso
l'indirizzo IP linklocal del vicino per ogni arco che lo stesso modulo ha realizzato.

**sistema **
```
ip route add 169.254.119.176 dev eth1 src 169.254.253.216
```

Assumiamo che il sistema ** abbia solo una identit *<sub>0</sub>* che si trova in una diversa rete
*G<sub></sub>*.

Il modulo Identities ha creato l'arco-identit principale, cio猫 quello che collega le due identit
*<sub>0</sub>* e *<sub>0</sub>*, senza per questo aggiungere alcuna rotta, perch茅 per tale arco-identit
la rotta 猫 stata aggiunta dal modulo Neighborhood nel network namespace default del sistema **.

Ora assumiamo che *<sub>0</sub>* decide di entrare in *G<sub></sub>*. Per essere precisi, il sistema ** decide di
costruire una nuova identit *<sub>1</sub>* partendo da *<sub>0</sub>*. L'identit *<sub>0</sub>* verr temporaneamente
spostata sul network namespace "entr01" del sistema **. Poi *<sub>1</sub>* far ingresso nella rete *G<sub></sub>*.

Quando il sistema ** crea la nuova identit, il suo modulo Identities dialoga con il modulo del sistema
** per aggiungere l'arco-identit *<sub>0</sub>-<sub>1</sub>* e modificare i valori (peer_mac e
peer_linklocal) dell'arco-identit *<sub>0</sub>-<sub>0</sub>*. Di fatto, questo comporta che il modulo Identities
nel sistema ** aggiunge la nuova rotta, sempre nel network namespace default gestito da *<sub>0</sub>*, verso il nuovo indirizzo
linklocal assunto da *<sub>0</sub>*.

**sistema **
```
ip route add 169.254.101.161 dev eth1 src 169.254.253.216
```

Ora il sistema ** fa entrare *<sub>1</sub>* in *G<sub></sub>* e, contemporaneamente, il sistema ** comunica
alla sua identit *<sub>0</sub>* che sull'arco-identit *<sub>0</sub>-<sub>1</sub>* va costruito un QspnArc.

Questo nuovo QspnArc che viene comunicato al modulo QSPN del sistema ** per l'identit *<sub>0</sub>*, inizialmente
non comporta operazioni sulle tabelle di routing, fino a quando non si riceve il primo ETP attraverso questo
arco e cos矛 si scopre l'indirizzo Netsukuku di questo vicino.

Dopo un po' di tempo il nodo *<sub>0</sub>* ricever un ETP dal QspnArc *<sub>0</sub>-<sub>1</sub>* e con esso
aggiorner l'indirizzo Netsukuku di questo vicino. Questo deve produrre due macro-operazioni nel sistema **: la creazione di
una tabella per i pacchetti IP ricevuti dall'arco *<sub>0</sub>-<sub>1</sub>* e l'aggiornamento (su tutte le tabelle) delle
rotte che adesso possono avere come gateway l'arco *<sub>0</sub>-<sub>1</sub>*.
